1. THREAT HUNTING
--------------------------------------------------

Hypothesis:
Suspicious SSH login attempts may indicate unauthorized access attempts.

Methodology:
Threat hunting was performed using Wazuh SIEM. Logs were analyzed using queries such as "sshd" and "rule.id:5710".

Findings:
Multiple failed login attempts were detected for invalid users such as "fakeuser" and "wronguser". These attempts were repeated several times, indicating a brute-force attack.

MITRE ATT&CK:
T1110.001 (Password Guessing)
T1021.004 (SSH)

Conclusion:
The system experienced repeated failed login attempts, indicating a potential brute-force attack.

--------------------------------------------------
2. SOAR PLAYBOOK
--------------------------------------------------

Scenario:
SSH brute-force attack detected.

Playbook Steps:
1. Detect Alert – Wazuh detects failed login attempts
2. Check IP – Identify source IP (::1)
3. Validate Threat – Confirm repeated failures
4. Block IP – Block suspicious IP
5. Create Ticket – Record incident
6. Notify Admin – Send alert notification

Playbook Table:

Playbook Step       | Action Taken
---------------------------------------------
Detect Alert        | Wazuh alert triggered

Notify Admin        | Alert sent to admin

Summary:
This SOAR playbook automates response to SSH brute-force attacks and improves response time.

3. POST INCIDENT ANALYSIS
--------------------------------------------------

Incident:
SSH brute-force attack detected.

Root Cause Analysis (5 Whys):
Why attack happened? → Weak authentication
Why weak? → No login restrictions
Why no restrictions? → Not configured
Why not configured? → Lack of security policies
Why? → Poor security management

Root Cause:
Weak authentication controls and lack of monitoring.

Lessons Learned:
Stronger authentication controls and monitoring are required.

Metrics:
MTTD: 1 hour
MTTR: 2 hours

Conclusion:
Proper security configuration can prevent such attacks.
